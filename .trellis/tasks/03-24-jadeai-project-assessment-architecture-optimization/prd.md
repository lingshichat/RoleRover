# JadeAI 求职场景 AI 体验优先级梳理

## Goal

从真实求职投递场景出发，明确 JadeAI 在“AI 帮我写简历、改简历、针对目标 JD 定制简历”这条主路径上的产品优先级，并给出可以直接指导后续实施的 P0 / P1 / P2 结论。

## Requirements

- 结论必须贴合个人求职与海投场景，而不是泛化评估整个项目
- 优先级判断围绕“写得更好、改得更快、针对岗位更准”展开
- P0 聚焦“目标 JD 定制闭环”
- P0 需要支持同一份简历针对多个 JD 生成不同版本
- JD 版本优先采用“独立简历副本”而不是复杂版本树
- JD 版本由用户手动创建，避免误改主简历
- JD 版本识别采用“标题 + JD 标签信息”
- JD 标签以岗位名为主，公司名可选，适配海投
- JD 优化建议按模块呈现
- 模块建议支持多选后批量应用
- 应用后自动重新分析并反馈最新结果

## Acceptance Criteria

- [x] 能在 5 分钟内读懂当前任务目标与优先级结论
- [x] 已形成围绕 AI 简历写作体验的 P0 / P1 / P2 清单
- [x] 每个优先级项都能说明解决什么真实使用问题
- [x] 已明确后续最值得先做的方向是 P0 的 JD 定制闭环

## Definition of Done

- AI 主路径、核心问题和优先级已经明确
- PRD 足够精简，不再膨胀成技术备忘录
- 结果能直接指导下一步产品设计与实现拆解

## Technical Approach

### Priority Summary

#### P0

目标 JD 定制闭环。

- 手动基于当前简历创建一个 JD 版本副本
- 版本以岗位名优先、公司名可选的 JD 标签识别
- 对该版本运行 JD 分析
- 按模块给出优化建议，例如 `Summary`、`Work Experience`、`Skills`
- 支持勾选多个模块建议后批量应用
- 应用后自动重新分析，展示最新匹配分、缺失关键词和后续建议

#### P1

AI 共写协作体验增强。

- 强化编辑器内聊天在长会话下的连续性
- 降低 AI 直接改写带来的失控感
- 让聊天改写、语法检查、JD 优化在交互上更一致

#### P2

基于真实经历生成更可信的 AI 初稿。

- 降低从零生成时的模板味和幻觉问题
- 强化“输入真实经历 -> 生成可信初稿”的质量

### Implementation Plan

- PR1: 设计 JD 版本副本的创建入口、命名规则和标签展示方式
- PR2: 设计 JD 分析结果页的模块建议卡片与多选批量应用交互
- PR3: 打通“批量应用 -> 自动重新分析 -> 反馈结果”的闭环
- PR4: 回头补强 P1 的 AI 共写体验，包括长对话稳定性与更可控的改写反馈

## Decision (ADR-lite)

**Context**

用户在真实使用里最常依赖的是 AI 帮写和帮改简历，而当前项目虽然已经具备聊天、整份生成、JD 分析、语法检查、翻译等能力，但这些入口较分散，对“我要为某个岗位快速打磨一份可投递版本”这条主路径支持还不够强。

**Decision**

将 P0 主路径确定为“目标 JD 定制闭环”，并采用以下产品选择：

- 同一份简历针对多个 JD 的版本，优先使用独立简历副本承载
- 副本由用户手动创建
- 版本识别以岗位名为主，公司名可选
- JD 优化建议按模块呈现
- 建议支持多选后批量应用
- 应用后自动重新分析，形成闭环

**Consequences**

- 优先强化可控性和可审阅性，而不是一次性自动改完整份简历
- 更适合海投和高频定制场景
- 能最大程度保护主简历不被误改
- 由于当前数据模型没有版本关系字段，P0 更适合先做轻量副本方案
- 后续如果验证价值足够，再考虑更重的版本关系或投递工作区模型

## Out of Scope

- 在本任务中修改 `src/` 下业务代码
- 在本任务中完成大文件拆分
- 在本任务中设计完整版本树系统
- 在本任务中穷举所有 AI 细节和所有产品能力
- 在本任务中评估与 AI 简历写作无关的全部功能

## Technical Notes

- 当前 AI 入口分散在聊天、整份生成、JD 分析、语法检查、翻译、求职信等多个位置，用户目标更像一条工作流而不是一组工具
- 当前 `/api/ai/chat` 只保留最近 10 轮、20 条消息进入模型上下文，长会话可能失忆
- 当前 JD 分析虽然已有评分、缺失关键词和 suggestions，但“优化”动作还是通过 `pendingAiMessage` 回到聊天面板完成，闭环不够直接
- 当前项目已经具备多份简历管理与 `/api/resume/[id]/duplicate` 能力，适合作为 P0 的 JD 版本副本基础
- 当前仪表盘卡片和列表已有标题、Badge、最近编辑时间，适合承载“标题 + JD 标签”的轻量版本识别方式
